# MVP — что сделано и что осталось

Дата: 2026-06-18. Документ отражает фактическое состояние после сборки скелета и
запуска в Docker. Объём MVP определён в [`../CONTEXT.md`](../CONTEXT.md) §13.

---

## ✅ Сделано и проверено вживую

- **Архитектура целиком:** модульный монолит (5 модулей с чистыми ацикличными
  границами), ADR-0001…0008.
- **Бэкенд работает в Docker:** Postgres + async SQLAlchemy + Alembic-миграция
  применена, идемпотентный сидер залил демо-контент.
- **Сквозной поток MVP доказан через API:** регистрация → вход (JWT access +
  refresh-cookie) → карта с линейным гейтингом → урок → тест (грейдинг на сервере,
  100%) → тренажёр (серверный грейдинг) → **блок закрыт → +30 XP → уровень 2** →
  следующий блок разблокирован.
- **Юнит-тесты ядра:** 9 тестов на грейдинг (тесты/тренажёр/допуски) — зелёные.
- **Фронтенд собирается и отдаётся:** Vite+React+TS+Mantine, `tsc --noEmit` чисто,
  feature-структура, адаптер Fortune-sheet, страницы карты/курса/теста/тренажёра/
  профиля/входа.
- **Тренажёр укреплён по P0-пунктам 2–4:** адаптер читает актуальную модель через
  `getSheet()`, явно форматирует числовые ячейки, фиксирует не-китайскую локаль/
  валюту, блокирует не-editable ячейки, а грейдинг стал лояльнее к форматированным
  числам и формулам без начального `=`. `PMT`/`IPMT`/`PPMT` подтверждены smoke-тестом
  установленного Fortune formula-parser.

---

## ⛏️ Осталось до «полного» MVP

Приоритеты: **P0** — без этого MVP неполноценен; **P1** — важно, но не блокирует
демо; **P2** — полировка.

### P0 — функциональные дыры

2. **Финальный browser-smoke тренажёра на поднятом backend+seed.**
   Адаптер покрыт Vitest: чтение актуального `data/celldata`, сбор `value`+`formula`,
   блокировка `editable`, числовой формат и наличие `PMT`/`IPMT`/`PPMT` в движке.
   Остаётся ручной/браузерный smoke полного рантайма: открыть задачу, ввести формулу,
   убедиться, что submit приводит к `passed: true` и закрывает шаг на карте.
   *Готово, когда:* решение задачи в UI приводит к `passed: true`.

### P1 — тесты и CI (по плану тестирования из CONTEXT §11)

5. **Integration-тесты API (по эндпоинту на модуль).**
   Нужна DB-backed фикстура: Postgres через testcontainers + откат транзакции на
   тест. Каркас-заметка уже в `backend/tests/conftest.py`. *Минимум:* регистрация/
   логин, выдача теста без правильных ответов, грейдинг теста и тренажёра, карта.

6. **1–2 сквозных backend-теста** на поток «урок→тест→тренажёр→закрытие блока→XP/
   уровень» (то, что сейчас проверено руками через скрипт — закрепить тестом).

7. **Фронтенд-тесты (Vitest + RTL).**
   Конфиг и `setup.ts` готовы, но тестов нет. *Минимум:* рендер карты со статусами,
   адаптер тренажёра (A1↔RC-конвертация, сбор CellModel), прохождение теста.

8. **CI (GitHub Actions).**
   Воркфлоу на PR: линт (`ruff`) + бэк-тесты (с сервисом Postgres) + фронт
   (`tsc --noEmit`, `vitest`). Файла `.github/workflows/ci.yml` пока нет.

### P2 — UX и контракт

9. **Рендер Markdown уроков.** Сейчас сырой текст (`whiteSpace: pre-wrap`).
   Добавить `react-markdown`. *Где:* `frontend/src/features/courses/LessonPage.tsx`.

10. **Генерация типов из OpenAPI.** Скрипт `npm run generate:types` готов; сейчас
    типы фронта написаны вручную. Сгенерировать `src/shared/api/schema.d.ts` и
    перевести feature-api на него (ADR-0008).

11. **Кнопка выхода и гард приватных маршрутов.** `logout()` в api есть, кнопки в
    UI нет; `/profile` доступен без входа (показывает заглушку). Добавить выход в
    шапку и редирект неавторизованных на `/login`.

12. **Авто-рефреш access-токена при 401.** Обёртка `apiFetch` сейчас не пытается
    обновить токен по истечении — добавить однократный retry через `/auth/refresh`.

13. **Удобство первого запуска.** Сейчас миграция/сид — ручные шаги (README).
    Можно добавить entrypoint-скрипт бэкенда `alembic upgrade head && seed` (но
    начальную миграцию всё равно генерируют один раз вручную).

---

## 🚫 Вне MVP (отложено осознанно)

Админка/CMS, email-верификация, OAuth, сброс пароля, достижения/бейджи,
лидерборды, ветвление карты, диаграммы/эксклюзивные фичи тренажёра, пересчёт
Excel-формул на сервере, мобильное приложение. (См. CONTEXT §13 и ADR-0004/0005/0007.)

---

## Рекомендуемый порядок

P0 (1→2) закрывает функциональную полноту демо; затем P1 (5–8) даёт защиту от
регрессий и CI; P2 — полировка. Пункт 2 — это ручная проверка тренажёра в браузере
на поднятом backend+seed.
