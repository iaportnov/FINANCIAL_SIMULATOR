# ADR-0001 — Модульный монолит с вертикальными слайсами и мягкими границами

Статус: принято · Дата: 2026-06-18

## Контекст
Требование: каждая сущность (профиль, курсы, тесты, тренажёр, прогресс) написана
отдельно, чтобы над ней можно было работать независимо. Команда маленькая, цель —
задать архитектуру + MVP.

## Решение
Один **модульный монолит** (FastAPI), организация кода — **вертикальные слайсы**
(package-by-feature): каждый модуль это пакет со своими
`router · service · repository · models · schemas` в `app/modules/<name>/`.

Одна общая БД (Postgres), но **каждый модуль владеет своими таблицами**; на чужие
сущности ссылаемся **по ID без межмодульных foreign keys**. Целостность держим в коде.

Межмодульное взаимодействие — прямой вызов публичных функций из `service.py` соседа,
зависимости строго однонаправленные (см. ADR-0003).

## Последствия
- (+) Модули реально независимы, легко делятся между людьми, безболезненно выносятся
  в сервис при росте.
- (+) Нет накладных расходов распределённой системы на MVP.
- (−) Часть целостности (межмодульные ссылки) не гарантируется БД — следим в коде/тестах.

## Альтернативы
- Layered (package-by-layer) — отвергнут: сущность размазана по слоям, тяжело
  работать над ней отдельно.
- Микросервисы — отвергнут: избыточно для MVP и маленькой команды.
