# ADR-0008 — Архитектура фронтенда

Статус: принято · Дата: 2026-06-18

## Контекст
React + TypeScript SPA. Тот же принцип модульности, что и на бэкенде: каждая сущность
разрабатывается отдельно. Контракт фронт/бэк не должен разъезжаться.

## Решение
- **Feature-based структура**, зеркалит бэкенд:
  `src/features/{profile,courses,quizzes,trainer,progression}` (components/api/hooks/
  types), `src/shared` (ui-kit, api-client, auth), `src/app` (роутинг, провайдеры).
- **React Router** — навигация.
- **TanStack Query** — серверный стейт (кэш, инвалидация, статусы).
- **Zustand** — лёгкий стор сессии (текущий пользователь, access-токен в памяти).
- **Mantine** — UI-кит.
- **Типы из OpenAPI** (`openapi-typescript`) + тонкая типизированная обёртка `fetch`.
- **Карта заданий** — свои React-компоненты (вертикальная «тропа» узлов со статусами
  из `progression`-API), без граф-библиотек (путь линейный на MVP).
- Тренажёр — внутри `features/trainer` живёт адаптер Fortune-sheet (см. ADR-0006).

## Последствия
- (+) Симметрия с бэкендом; типобезопасный контракт; минимум глобального стейта.
- (−) Генерация типов добавляет шаг в пайплайн (запуск при изменении API).

## Альтернативы
- Redux — избыточно; Tailwind+shadcn — больше ручной работы (см. ADR-0002).
- Граф-библиотека для карты — не нужна при линейном пути.
