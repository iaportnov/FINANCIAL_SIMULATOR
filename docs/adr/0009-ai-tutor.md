# ADR-0009 — ИИ-наставник в задачах тренажёра

Статус: принято · Дата: 2026-06-18

## Контекст
В задачах тренажёра учащемуся нужна помощь «здесь и сейчас»: объяснение стандарта,
наводящий вопрос, подсказка по методу расчёта. ИИ-наставник — часть ядра продукта
(слой B: grounded-тьютор). Ключевой риск — утечка ответа: если модель получит
секретный эталон грейдинга, она просто продиктует число, и тренажёр перестанет
учить. Подключение модели идёт через внешний API (Anthropic Claude).

## Решение
**Отдельный вертикальный модуль `tutor`** (`router · service · schemas · prompts ·
client`), как и остальные модули монолита. Зависимости направленные:
`tutor → trainer` (читает **публичную** проекцию задачи) и `tutor → users`
(`current_user`). Цикла нет: `trainer`/`progression` про `tutor` не знают.

**Жёсткий guardrail (главное решение).** Наставник получает только:
- публичную часть задачи (`instructions_md`, `sheet`, `editable`) — ту же, что уходит
  на клиент;
- текущее состояние ячеек учащегося, присланное клиентом;
- историю диалога (бэкенд stateless, переписка не хранится на MVP).

Секретный блок `grading` (эталоны, `must_use_function`) **не загружается** —
`tutor.service` ходит только в `TrainerRepository.get_task`, не в `get_grading`.
Системный промпт прямо запрещает называть итоговое число/готовую формулу и выносить
вердикт «верно/неверно» — это работа детерминированного грейдера. Так выполняется
продуктовый принцип: *ИИ не вычисляет авторитетное число и не формулирует
авторитетное правило*.

**Вендор за адаптером.** Anthropic SDK изолирован в `tutor/client.py` за нейтральным
протоколом `TutorClient` (system + реплики → текст). Смена провайдера — переписать
один файл (как движок тренажёра в ADR-0006). Импорт `anthropic` ленивый: приложение
и несвязанные тесты поднимаются без установленного пакета.

Конфиг — через `pydantic-settings`: `ANTHROPIC_API_KEY` (пустой ключ выключает
эндпоинт → `validation_error`), `TUTOR_MODEL` (по умолчанию `claude-opus-4-8`),
`TUTOR_MAX_TOKENS`. Эндпоинт `POST /trainer/{task_id}/assistant` требует
аутентификации (контроль стоимости).

## Последствия
- (+) Наставник «понимает» задачу из авторского контента, но архитектурно не может
  слить ответ — эталон ему недоступен.
- (+) Тонкий сквозной срез: бэкенд stateless, без новой таблицы и миграции.
- (+) Провайдер меняется в одном файле; ключ-переключатель включает/выключает фичу.
- (−) Контекст и переписку шлёт клиент каждый раз — токены не кэшируются, диалоги не
  аналитизируются. Путь роста: персистентность диалогов + спайс-повторение (слой B).
- (−) Подсказки идут из `instructions_md`; авторские `hints` пока в БД не хранятся
  (кандидат на следующий шаг — публичная колонка `hints`).

## Альтернативы
- **Грейдинг-эталон в контекст модели** — отклонено: ломает дидактику и анти-чит.
- **Эндпоинт под `/tutor`** вместо `/trainer/{id}/assistant` — путь оставлен под
  `trainer` для обнаружимости; модуль при этом отдельный.
- **Серверное хранение диалогов на MVP** — отложено ради тонкого среза.
