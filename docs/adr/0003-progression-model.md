# ADR-0003 — Модель прогресса: статичная программа vs динамика пользователя

Статус: принято · Дата: 2026-06-18

## Контекст
Поток обучения: `урок → тест → задача в тренажёре`, шаги сгруппированы в блоки, за
блок начисляется XP и растёт уровень. Карта заданий — последовательность блоков.
Нужно связать разнородные активности, не делая модули зависимыми циклически.

## Решение
Иерархия **Course → Block → Step → Activity**(`lesson|quiz|trainer_task`).

Разделяем статику и динамику по разным модулям:
- **`courses`** владеет статичной программой: `Course`, `Block`, `Step`, `Lesson`.
  `Step` ссылается на активность **полиморфно** (`activity_type + activity_id`),
  с денормализованным `title` для отрисовки карты.
- **`progression`** владеет динамикой пользователя: `ActivityCompletion`,
  `UserProgress(total_xp, level)`, `LevelThreshold`. Читает структуру из `courses`,
  принимает отчёты о завершении, начисляет XP за закрытые блоки, считает уровень.

**Отчёт о завершении — «снизу вверх»:** `quizzes` и `trainer` по завершении зовут
`progression.record_completion(...)`. Завершение **урока** (грейдинга нет)
обрабатывает сам `progression`. Поэтому `courses` ни от кого не зависит, и граф
ацикличен: `quizzes/trainer → progression → courses`, все → `users`.

Карта на MVP **строго линейная**; на `Course` заложен флаг `gating` на будущее.

## Замечание
Уточняет ранний набросок, где `Step` предполагался в модуле `progression`.
Перенесён в `courses` (статика) ради ацикличности зависимостей; полиморфная ссылка и
направление отчётов сохранены.

## Последствия
- (+) Чистое разделение «статичный контент / динамика пользователя», нет циклов.
- (+) Прогресс не знает внутренностей контентных модулей (только `type + id`).
- (−) `progression` читает структуру из `courses` — связь по сервисному интерфейсу.

## Альтернативы
- Шина событий вместо прямых вызовов — путь роста, отложено на MVP.
- Полностью разместить Course/Block/Step в `progression` — даёт цикл при завершении урока.
