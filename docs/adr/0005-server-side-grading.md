# ADR-0005 — Серверный грейдинг с секретным эталоном

Статус: принято · Дата: 2026-06-18

## Контекст
Тесты и задачи тренажёра нужно проверять. Важно: не раскрывать правильные ответы
клиенту и не давать «подсмотреть» эталон через DevTools/Network.

## Решение
**Вердикт выносит сервер.** Правильные ответы и правила грейдинга хранятся только в
БД и **никогда не сериализуются клиенту**.

- **Тесты:** клиент шлёт ответы → сервер считает балл, сверяет с порогом
  (`pass_threshold`, по умолчанию 70%), возвращает балл + разбивку верно/неверно
  (+ опциональные объяснения). Пересдачи без ограничений.
- **Тренажёр:** таблица считается на клиенте (UX/живой фидбэк); на «Проверить» клиент
  шлёт модель ячеек (значения + текст формул); сервер сверяет с секретным эталоном и
  правилами и возвращает только вердикт по ячейкам + балл.

**Гибридная проверка** (настраивается на уровне задачи): по умолчанию сверка
значений (с допуском для чисел), опционально — требование «это формула»
(`must_be_formula`) и/или «использована функция X» (`must_use_function`, толерантный
парсинг).

Пересчёт Excel-формул на сервере (Python) на MVP **не делаем** — доверяем значениям
от клиента и сверяем с секретным эталоном.

## Последствия
- (+) Эталон не утекает; «обмануть» = знать ответ = по сути решить.
- (+) Грейдинг не зависит от движка таблицы (сверяем нашу модель ячеек).
- (−) Теоретически клиент может прислать подделанное значение, но без знания эталона
  это бесполезно. Модель угроз для обучалки приемлема.

## Альтернативы
- Проверка на клиенте — отвергнуто: эталон утекает.
- Серверный пересчёт формул (`pycel`/`formulas`) — отложено как тяжёлая зависимость.
